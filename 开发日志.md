### 00:10 安装依赖 - 网络问题排查

**遇到的错误：** pip install 全部失败，报 `ConnectionResetError(10054)`

```
ERROR: Could not find a version that satisfies the requirement paddlepaddle
ProtocolError('Connection aborted.', ConnectionResetError(10054, ...))
```

**排查过程：**
1. 试了默认 PyPI → 失败
2. 试了百度镜像 `-i https://mirror.baidu.com/pypi/simple` → 失败
3. 试了清华镜像 `-i https://pypi.tuna.tsinghua.edu.cn/simple` → 失败
4. `ping pypi.org` → 能通（5ms），说明网络本身没问题
5. `curl -sv https://pypi.org/` → 发现 SSL 错误：
   ```
   schannel: next InitializeSecurityContext failed: CRYPT_E_NO_REVOCATION_CHECK (0x80092012)
   ```
6. 检查 Windows 代理设置（PowerShell）：
   ```
   ProxyEnable: 0（关闭状态）
   ProxyServer: 127.0.0.1:7897
   ```

**根因：** 电脑配置了 Clash 代理（127.0.0.1:7897），代理软件未开启时，系统的网络流量被拦截但无法转发，导致 HTTPS 连接被重置。

**解决方案：** 开启 Clash 代理，并在 pip 命令中通过环境变量指定代理：
```bash
export https_proxy=http://127.0.0.1:7897
export http_proxy=http://127.0.0.1:7897
pip install ...
```

### 00:20 安装 PaddlePaddle + PaddleOCR

**第一次安装：**
```bash
pip install paddlepaddle paddleocr fastapi python-multipart aiofiles
```
安装成功：paddlepaddle 3.3.0、paddleocr 3.4.0

**遇到的错误 1：** PaddleOCR 初始化报错
```python
PaddleOCR(use_angle_cls=True, lang='en', show_log=False)
# ValueError: Unknown argument: show_log
```
**原因：** PaddleOCR 3.4 是大版本更新，API 完全重构了，不再支持 `show_log` 参数。
**修复：** 删除 `show_log=False`

**遇到的错误 2：** OCR 推理报错
```python
ocr.ocr(image_path, cls=True)
# TypeError: PaddleOCR.predict() got an unexpected keyword argument 'cls'
```
**原因：** PaddleOCR 3.4 的 `ocr()` 方法不再接受 `cls` 参数。
**修复：** 删除 `cls=True`

**遇到的错误 3：** PaddlePaddle 推理引擎报错
```
NotImplementedError: ConvertPirAttribute2RuntimeAttribute not support [pir::ArrayAttribute<pir::DoubleAttribute>]
```
**原因：** PaddlePaddle 3.3.0 + PaddleOCR 3.4.0 的 oneDNN 后端有兼容性 bug，这是已知问题。

**最终解决方案：** 降级到稳定版本组合
```bash
pip install "paddlepaddle==2.6.2" "paddleocr==2.9.1"
```
降级后 `show_log` 和 `cls` 参数恢复可用，OCR 推理正常。

**经验教训：** PaddlePaddle/PaddleOCR 的大版本之间 API 变化很大，生产环境应锁定版本号。推荐的稳定组合：PaddlePaddle 2.6.x + PaddleOCR 2.9.x。

### 00:40 编写核心代码

**database.py** - 4 张表：
- `invoices` - 票据主表
- `fields` - 字段结果表
- `candidates` - 候选值表
- `audit_logs` - 审计日志表

**ocr_engine.py** - PaddleOCR 封装：
- `run_ocr(image_path)` - 对一张图跑 OCR，返回统一格式
- 单例模式避免重复加载模型（加载约需 2-3 秒）

**scorer.py** - 多候选评分算法（论文核心）：
- 评分公式：`S(i) = α * C_ocr + β * F_format + γ * X_cross`
- 格式校验函数：日期、金额、公司名、地址各一个
- 跨字段约束：金额>0、日期不超未来、字段不重复等
- 4 组消融配置：baseline / format / crossfield / full
- 决策规则：格式一票否决 → 评分排序 → 异常标记

**extractor.py** - 字段提取引擎：
- 针对 SROIE 的 4 个字段设计了提取策略
- company: 取收据顶部 + SDN BHD 等关键词匹配
- date: 正则匹配日期格式 + "date" 关键词
- address: 地址关键词行 + 多行合并
- total: "total" 关键词 + 右侧/下方金额 + RM 货币前缀

**data_loader.py** - 数据集加载和评估：
- 加载 SROIE 图片和 ground truth
- `match_field()` 字段匹配函数（支持完全/包含/去空格/数值匹配）

**run_ablation.py** - 消融实验运行器：
- 支持 `--max-samples` 参数（调试用）
- 支持 `--use-cache`（跳过 OCR 推理）
- 自动保存 CSV / Markdown / JSON 结果

### 01:40 GPU 版本切换 - 失败，退回 CPU

**尝试：** 安装 GPU 版 PaddlePaddle
```bash
pip install paddlepaddle-gpu==2.6.2
```
安装成功（477MB），GPU 检测正常：
```
GPU available: True
GPU name: NVIDIA GeForce RTX 3050 Laptop GPU
```

**遇到的错误：** OCR 推理时报错
```
RuntimeError: The third-party dynamic library (cudnn64_8.dll) that Paddle depends on is not configured correctly.
```
**原因：** PaddlePaddle 2.6.2 GPU 版是用 CUDA 11.8 + cuDNN 8 编译的，但系统上是 CUDA 12.3（自带 cuDNN 9）。Windows 上 PaddlePaddle 2.6.x 没有提供 CUDA 12 的预编译 wheel。

**解决方案选择：**
- 方案A: 下载 cuDNN 8 的 DLL 文件放到 CUDA 目录（可行但麻烦）
- 方案B: 用 CPU 版跑实验（不影响准确率，只是慢一些）

**决定：** 先用 CPU 版出数据。GPU 加速只影响速度，不影响实验结论。

```bash
pip uninstall paddlepaddle-gpu -y
pip install paddlepaddle==2.6.2
```

**额外问题：** 重装后 `import paddle` 报错 `ModuleNotFoundError`。原因是 GPU 版卸载时可能清掉了一些文件。解决：
```bash
pip install --force-reinstall paddlepaddle==2.6.2
```

### 01:50 NumPy 兼容性问题

**遇到的错误：**
```
AttributeError: `np.sctypes` was removed in the NumPy 2.0 release.
```
**原因：** `--force-reinstall` 把 NumPy 升级到了 2.4.3，而 PaddleOCR 依赖的 `imgaug` 库使用了 NumPy 1.x 的 `np.sctypes` API，该 API 在 NumPy 2.0 中被删除。

**解决：**
```bash
pip install "numpy<2.0"  # 降级到 1.26.4
```

**经验教训：** `pip install --force-reinstall` 会重装所有依赖，可能导致版本变化。如果只是修复某个包，优先用 `pip install --force-reinstall --no-deps`。

### 03:00 消融实验优化

**问题：** 第一版消融实验四组结果几乎没有差异（Format 组 = Baseline，Full = CrossField）

**根因分析：**
- 每个字段候选值太少（address 永远只有1个候选）→ 评分算法没东西可选
- 格式校验是二值的（0或1）→ 几乎所有候选都通过 → 没有区分度
- 跨字段约束太少 → 对 total 和 date 的区分不够

**优化方案：**

1. **生成更多候选值：**
   - Company: 取顶部 8 个文本块 + 全文关键词搜索
   - Date: 日期格式匹配 + 含数字的短文本块（"伪日期"让格式校验来排除）
   - Address: 多种合并策略（全范围/核心行/单行/双行）
   - Total: 收据上所有金额数值（小计/税额/总额），让跨字段约束筛选最大金额

2. **格式校验改为连续分数（0~1）：**
   - Date: 严格格式 1.0 → 宽松格式 0.7 → 含数字 0.1 → 无数字 0.0
   - Total: `xx.xx` 格式 1.0 → 整数 0.6 → 其他小数 0.4 → 含数字 0.1
   - Company: 含公司关键词 +0.4，合理长度 +0.2，含字母 +0.1
   - Address: 含地址关键词 +0.3，含邮编 +0.2，含州名 +0.15

3. **增强跨字段约束：**
   - Total 应是收据中最大金额（>=90%最大值得满分）
   - Total 不应太小（<1.00 的总额罕见）
   - Date 年份应在 2000~2030 范围内
   - Company 不应像日期或金额

**优化后全量结果（626张）：**

| 实验组 | α | β | γ | 总准确率 | Company | Date | Address | Total |
|--------|---|---|---|----------|---------|------|---------|-------|
| Baseline | 1.0 | 0 | 0 | 56.87% | 29.71% | 81.47% | 53.83% | 62.46% |
| +Format | 0.6 | 0.4 | 0 | 63.34% | 51.12% | 85.94% | 53.83% | 62.46% |
| +CrossField | 0.5 | 0 | 0.5 | 60.22% | 29.87% | 90.42% | 53.67% | 66.93% |
| **Full Model** | 0.4 | 0.3 | 0.3 | **65.50%** | **51.12%** | **90.26%** | 53.67% | **66.93%** |

Full Model vs Baseline 提升 **+8.63%**，各分量贡献清晰。

### 性能测试与可解释性指标 (Option B)

**做了什么：**
1. 创建 `run_performance_test.py`，整合性能测试和可解释性指标计算
2. 使用缓存的 OCR 结果（之前消融实验已缓存），对全部 626 张图运行测试
3. 测量的指标：
   - **性能**: OCR 时间、字段提取时间、总处理时间、内存峰值、吞吐量
   - **可解释性**: ECR（证据覆盖率）、DTS（决策透明度）、TCS（追溯完整度）
4. 使用 `tracemalloc` 测量内存占用
5. 每张图都写入 SQLite 数据库（invoices + fields + candidates + audit_logs），验证 TCS

**遇到的问题：**
- `sqlite3.ProgrammingError: type 'WindowsPath' is not supported` → `sample["image_path"]` 是 PathLib 对象，需要 `str()` 转换

**可解释性指标定义：**
- **ECR = N_mapped / N_total**: 有视觉证据坐标(bbox)的字段数 / 总字段数
- **DTS = N_explained / N_total**: 有完整决策信息（候选列表+评分明细+决策原因）的字段数 / 总字段数
- **TCS = N_traceable / N_total**: 有完整审计日志（ocr_complete + extraction_complete）的票据数 / 总票据数

**性能测试结果：**

| 测试指标 | 平均值 | 中位数 | 标准差 | 最小值 | 最大值 |
|----------|--------|--------|--------|--------|--------|
| OCR识别时间(ms) | 6142.8 | 5714.3 | 2310.0 | 1101.1 | 16125.1 |
| 字段提取时间(ms) | 9.5 | 8.0 | 5.9 | 3.8 | 64.8 |
| 单张总处理时间(ms) | 10.9 | 9.2 | 6.7 | 4.5 | 71.5 |

- 端到端吞吐量(含OCR): ~9.8 张/分钟 (CPU模式)
- 内存峰值(提取阶段): 0.2 MB

**可解释性指标结果：**

| 指标 | 测量值 | 目标值 | 达标 |
|------|--------|--------|------|
| ECR | 99.96% | ≥95% | Yes |
| DTS | 99.96% | ≥90% | Yes |
| TCS | 100.00% | ≥95% | Yes |

三个可解释性指标均超过目标值。

- `experiments/performance_report.md` — Markdown 报告（可直接引用到论文）
- `experiments/performance_results.json` — JSON 完整数据
- `experiments/performance_results.csv` — CSV 格式
